# Introduction to Java 8

### 🔹 Why Java 8?

Before Java 8, developers faced some problems:

1. **Verbose Code** – Writing anonymous classes (like Runnable, Comparator) required too many lines.

   ```java
   Runnable r = new Runnable() {
       @Override
       public void run() {
           System.out.println("Running...");
       }
   };
   ```

   → Same work in Java 8 with **lambda**:

   ```java
   Runnable r = () -> System.out.println("Running...");
   ```

2. **Poor support for Functional Programming** – Java was purely OOP. But modern programming also needs **functional style** (passing functions as parameters, immutability, parallelism).

3. **Collections Processing was Painful** – Before Java 8, iterating and filtering data meant writing multiple for-loops. Example: find names starting with “A” from a list.

   * Old way: verbose for-loops.
   * Java 8 way: **Streams API** (`list.stream().filter(name -> name.startsWith("A")).collect(...)`)

4. **Date & Time API was Broken** – `java.util.Date` and `java.util.Calendar` were:

   * Not thread-safe
   * Confusing (month starts from 0 😓)
   * Difficult to format/parse
     → Java 8 introduced `java.time` API (inspired by Joda-Time).

5. **Better Concurrency Needs** – Old `Future` was limited (no chaining, no combining results). Java 8 brought **CompletableFuture**.

---

### 🔹 Key Features Introduced in Java 8

* **Lambda Expressions** → Functional style programming
* **Functional Interfaces** → Predicate, Function, Consumer, Supplier, etc.
* **Method & Constructor References**
* **Streams API** → Easy collection processing
* **Collectors API** → Powerful grouping, partitioning, reduction
* **Optional Class** → To avoid `NullPointerException`
* **New Date & Time API** (`LocalDate`, `LocalTime`, `ZonedDateTime`)
* **Default & Static methods in Interfaces**
* **Nashorn JavaScript Engine** (later removed in Java 15, but interview point)
* **CompletableFuture** and concurrency improvements
* **Minor utilities**: StringJoiner, Base64 API, collection improvements

---

### 🔹 Backward Compatibility

* Java 8 maintained **100% backward compatibility**.
* Old Java 7/6/5 code runs fine on Java 8 **without modification**.
* New features are **additive**, not breaking. Example:

  * Interfaces now support `default` methods → does not break old classes implementing them.
  * Old `Date` class still exists, but new `java.time` API is preferred.
* JVM still runs `.class` files generated by older Java versions.

---

👉 In short:
Java 8 = **Turning point** in Java’s history → from pure OOP → **OOP + Functional Programming Hybrid**, with better productivity and modern APIs.

## 1. Introduction to Java 8 (Detailed Explanation)

Java 8 was a landmark release in the history of Java. Until Java 7, the language was mainly object-oriented and often criticized for being verbose and outdated compared to newer languages like Scala, Groovy, or even JavaScript. Developers were forced to write large amounts of boilerplate code for simple operations, such as iterating through collections, filtering data, or creating anonymous classes for functional behavior.

The release of Java 8 in **March 2014** fundamentally changed the way developers write Java programs. It introduced **functional programming concepts** into Java while maintaining its strong object-oriented foundation. This hybrid approach gave developers the ability to write cleaner, shorter, and more expressive code, without sacrificing Java’s stability and backward compatibility.

### Why Java 8 Was Needed

Before Java 8, there were several long-standing pain points:

1. **Verbosity of Anonymous Classes**
   Simple tasks like creating a thread or comparator required many lines of code. For example, implementing a `Runnable` interface meant overriding its method using an anonymous inner class, which cluttered the code. Java 8 introduced **lambda expressions**, which allow writing such behavior in a single line, making code more concise and readable.

2. **Lack of Functional Programming**
   Modern programming often benefits from functional paradigms, such as treating functions as first-class citizens, immutability, and declarative style. Languages like Scala or Kotlin already supported this, and Java was falling behind. Java 8 addressed this by introducing **functional interfaces** and **lambdas**, enabling developers to pass behavior (functions) as data.

3. **Cumbersome Collection Processing**
   Working with collections (Lists, Sets, Maps) required repetitive loops, conditions, and temporary variables. For instance, filtering a list of names or transforming data involved several steps. Java 8 introduced the **Streams API**, allowing developers to chain operations like `map`, `filter`, `reduce`, and process data in a declarative, pipeline style, often in a single line.

4. **Broken Date and Time API**
   The old `java.util.Date` and `Calendar` classes were poorly designed, not thread-safe, and confusing (for example, months were zero-based, leading to errors). Java 8 introduced the new `java.time` package, inspired by the popular Joda-Time library, which provided a clean, immutable, and thread-safe API for handling dates and times.

5. **Improved Concurrency Needs**
   Applications were increasingly moving toward parallelism and asynchronous processing. The old `Future` class was very limited, as it didn’t allow chaining tasks or combining multiple asynchronous results. Java 8 introduced **CompletableFuture**, making it easier to write non-blocking, asynchronous programs in a more fluent way.

---

### Key Features Introduced in Java 8

Some of the major additions in Java 8 include:

* **Lambda Expressions** – Introduced functional programming style by allowing functions to be passed as parameters.
* **Functional Interfaces** – Interfaces with a single abstract method (e.g., Predicate, Function, Consumer, Supplier).
* **Method and Constructor References** – Shortcuts to refer to existing methods or constructors.
* **Streams API** – Declarative processing of collections with support for parallelism.
* **Collectors API** – Grouping, partitioning, reducing, and summarizing data from streams.
* **Optional Class** – A container to avoid `NullPointerException` and represent optional values safely.
* **New Date and Time API** – A modern, thread-safe, immutable date and time system.
* **Default and Static Methods in Interfaces** – Allowed interfaces to evolve without breaking existing implementations.
* **Nashorn JavaScript Engine** – Enabled executing JavaScript code on the JVM.
* **CompletableFuture** – Powerful asynchronous programming construct.
* **Utility Improvements** – New classes like StringJoiner, Base64 API, and small updates in collections.

---

### Backward Compatibility

One of Java’s biggest strengths has always been backward compatibility, and Java 8 upheld this tradition. All existing Java code written in older versions continued to run on Java 8 without modification. New features were **additive**, not breaking:

* Interfaces gained `default` methods, but old classes that implemented them still worked fine.
* The old `Date` and `Calendar` APIs were not removed; they still function, though developers are encouraged to use the new `java.time` API.
* JVM remained compatible with `.class` files compiled from earlier versions.

This meant developers could gradually adopt Java 8 features in their projects without fear of breaking legacy systems.

---

✨ In summary, Java 8 was a **revolutionary upgrade**. It brought functional programming to Java, modernized its core APIs, and made it competitive with newer languages while preserving its stability and compatibility. It effectively transformed Java into a hybrid language that balances both **object-oriented** and **functional** paradigms.

## 2. Lambda Expressions (Detailed Explanation)

### 🔹 Background

Before Java 8, if you wanted to represent a piece of behavior (like a function) and pass it around, you had no direct way. You were forced to use **anonymous inner classes**. This led to long, boilerplate code even for simple tasks like sorting, filtering, or creating a thread.

For example, suppose we want to start a thread:

```java
// Before Java 8
Thread t = new Thread(new Runnable() {
    @Override
    public void run() {
        System.out.println("Running thread...");
    }
});
t.start();
```

This is too verbose for such a small task. Java 8 solved this by introducing **lambdas**, which let us represent functionality as an expression.

---

# Lambda Expression

A **lambda expression** is essentially a **short block of code** (function) that you can pass around and execute later.
In simple words: it’s a **function without a name** (an anonymous function).

**General Syntax:**

```java
(parameters) -> expression
```

or

```java
(parameters) -> { statements; }
```

Examples:

```java
() -> System.out.println("Hello World");            // no parameter
x -> x * x                                          // single parameter
(x, y) -> { return x + y; }                         // multiple parameters
```

---

### 🔹 Features of Lambdas

1. **Concise** – Much shorter than anonymous classes.
2. **Functional** – You can pass behavior as a parameter to methods.
3. **Type Inference** – Java infers parameter types from context, so you often don’t need to declare them.
4. **Integration with Functional Interfaces** – Lambdas can be used wherever a **functional interface** is expected.

---

### 🔹 Lambda with Functional Interface

A lambda expression can only be used with a **functional interface** (an interface with exactly one abstract method).

Example: Using `Runnable` interface

```java
Runnable r = () -> System.out.println("Running thread...");
new Thread(r).start();
```

Example: Using `Comparator`

```java
List<String> list = Arrays.asList("Banana", "Apple", "Mango");

// Before Java 8
Collections.sort(list, new Comparator<String>() {
    @Override
    public int compare(String s1, String s2) {
        return s1.compareTo(s2);
    }
});

// With Lambda
Collections.sort(list, (s1, s2) -> s1.compareTo(s2));
```

---

### 🔹 Types of Lambda Parameters

1. **No parameter**

   ```java
   () -> System.out.println("Hello!");
   ```
2. **One parameter**

   ```java
   name -> System.out.println("Hello " + name);
   ```
3. **Multiple parameters**

   ```java
   (a, b) -> a + b;
   ```

---

### 🔹 Return Type

* If the body has a **single statement**, `return` is not required.
  Example:

  ```java
  (a, b) -> a + b
  ```
* If the body has **multiple statements**, use `{ }` and explicitly `return`.
  Example:

  ```java
  (a, b) -> { 
      System.out.println("Adding numbers");
      return a + b;
  }
  ```

---

### 🔹 Scope and Access Rules ----Needs to check

* A lambda can access variables from its **enclosing scope** (like an inner class).
* But those variables must be **effectively final** (i.e., their value does not change).

Example:

```java
String greeting = "Hello"; // effectively final

Runnable r = () -> System.out.println(greeting + " Lambda");
r.run();
```

If you modify `greeting` later, it will give a compilation error.

---

### 🔹 Benefits of Lambda Expressions

1. **Less boilerplate code** → cleaner, shorter programs.
2. **Improved readability** → intent is more obvious.
3. **Encourages functional programming** in Java.
4. **Great with Collections** – sorting, filtering, mapping, reducing.
5. **Parallelism made easier** (with Streams + Lambdas).

---

### 🔹 Limitations of Lambda Expressions

1. They can only be used with **functional interfaces**.
2. Debugging can sometimes be tricky (stack traces show “lambda” instead of class names).
3. Overuse can reduce readability if lambdas are too complex.
4. Cannot declare checked exceptions directly in lambdas unless the functional interface allows them.

---

✨ In summary, **Lambda Expressions** make Java more expressive and less verbose. They enable Java to treat functionality as data, making it closer to modern functional languages while keeping its OOP base intact.

---

# Functional Interfaces Explaination

### 🔹 Background

In Java, interfaces have always been used to define contracts. But before Java 8, most interfaces had multiple abstract methods, so they could not directly support **lambda expressions**.

Lambda expressions need a **target type** to map to. That’s where **functional interfaces** come in. A functional interface is simply an interface with **exactly one abstract method (SAM – Single Abstract Method)**.

Because there’s only one abstract method, Java can **infer** which method the lambda expression should implement.

---

### 🔹 Definition

A **Functional Interface** is an interface that has **only one abstract method**, though it can have multiple `default` or `static` methods.

Examples:

* `Runnable` → `run()`
* `Callable` → `call()`
* `Comparator` → `compare()`
* `ActionListener` → `actionPerformed()`

---

### 🔹 `@FunctionalInterface` Annotation

Java 8 introduced `@FunctionalInterface` to clearly declare an interface as functional.

Example:

```java
@FunctionalInterface
interface MyFunctionalInterface {
    void greet(String name);

    // Allowed: default methods
    default void sayHello() {
        System.out.println("Hello!");
    }

    // Allowed: static methods
    static void info() {
        System.out.println("This is a functional interface");
    }
}
```

If you try to add a second abstract method, the compiler will throw an error.

---

### 🔹 Using Functional Interface with Lambda

```java
@FunctionalInterface
interface Calculator {
    int add(int a, int b);
}

public class Test {
    public static void main(String[] args) {
        Calculator calc = (a, b) -> a + b;
        System.out.println(calc.add(5, 10)); // Output: 15
    }
}
```

Here, the lambda `(a, b) -> a + b` is mapped to the `add()` method of `Calculator`.

---

### 🔹 Built-in Functional Interfaces (java.util.function package)

Java 8 provides many pre-defined functional interfaces to save effort:

1. **Predicate<T>** → `test(T t): boolean`

   * Represents a condition (returns true/false).
     Example:

   ```java
   Predicate<Integer> isEven = n -> n % 2 == 0;
   System.out.println(isEven.test(10)); // true
   ```

2. **Function<T,R>** → `apply(T t): R`

   * Takes one input and returns one output.
     Example:

   ```java
   Function<String, Integer> lengthFunc = s -> s.length();
   System.out.println(lengthFunc.apply("Danish")); // 6
   ```

3. **Consumer<T>** → `accept(T t): void`

   * Performs an operation but returns nothing.
     Example:

   ```java
   Consumer<String> print = s -> System.out.println(s);
   print.accept("Hello World");
   ```

4. **Supplier<T>** → `get(): T`

   * Returns a value without input.
     Example:

   ```java
   Supplier<Double> randomSupplier = () -> Math.random();
   System.out.println(randomSupplier.get());
   ```

5. **BiFunction<T,U,R>** → `apply(T t, U u): R`

   * Takes two inputs and returns one output.
     Example:

   ```java
   BiFunction<Integer, Integer, Integer> sum = (a, b) -> a + b;
   System.out.println(sum.apply(5, 7)); // 12
   ```

6. **BiPredicate<T,U>** → `test(T t, U u): boolean`

7. **BiConsumer<T,U>** → `accept(T t, U u)`

These interfaces make lambda programming very powerful.

---

### 🔹 Custom Functional Interface

You can also define your own functional interfaces when the built-in ones don’t fit.

Example:

```java
@FunctionalInterface
interface StringFormatter {
    String format(String s);
}

public class Test {
    public static void main(String[] args) {
        StringFormatter upper = str -> str.toUpperCase();
        System.out.println(upper.format("java8")); // JAVA8
    }
}
```

---

### 🔹 Advantages of Functional Interfaces

1. Provide target types for lambda expressions.
2. Allow functional programming in Java.
3. Increase code reusability with built-in `Predicate`, `Function`, etc.
4. Help in designing cleaner APIs.

---

### 🔹 Key Rules

1. Only **one abstract method** allowed.
2. Can contain multiple `default` and `static` methods.
3. `@FunctionalInterface` is optional but recommended.
4. <mark>Inherits abstract method from `Object` (like `toString()`) → still considered functional.</mark>

---

✨ In summary:
Functional Interfaces are the **foundation of lambda expressions** in Java 8. They let you treat behavior as data, making Java more expressive and closer to functional programming paradigms.

# Method and Constructor References

### 🔹 Background

When we use a lambda expression, sometimes its body just calls an existing method. For such cases, Java 8 provides **method references** — a shorthand syntax to refer to a method directly instead of writing a lambda.

Similarly, we can also reference constructors using the same syntax.

---

### 🔹 What is a Method Reference?

A **method reference** is a way to refer to an existing method by its name, without executing it. It is basically a **shortcut for a lambda expression** that calls that method.

General Syntax:

```
ClassName::methodName
```

For constructors:

```
ClassName::new
```

---

### 🔹 Types of Method References

There are 4 types of method references in Java 8:

1. **Reference to a static method**
   Syntax: `ClassName::staticMethodName`
   Example:

   ```java
   Function<String, Integer> func = Integer::parseInt;
   System.out.println(func.apply("123")); // Output: 123
   ```

   Equivalent Lambda:

   ```java
   Function<String, Integer> func = s -> Integer.parseInt(s);
   ```

---

2. **Reference to an instance method of a particular object** --Need to check difference between point 2 and 3
   Syntax: `instance::methodName`
   Example:

   ```java
   String message = "Hello";
   Supplier<String> supplier = message::toUpperCase;
   System.out.println(supplier.get()); // Output: HELLO
   ```

   Equivalent Lambda:

   ```java
   Supplier<String> supplier = () -> message.toUpperCase();
   ```

---

3. **Reference to an instance method of an arbitrary object of a particular type**
   Syntax: `ClassName::methodName`
   Example:

   ```java
   List<String> list = Arrays.asList("apple", "banana", "mango");
   list.forEach(System.out::println);
   ```

   Here, each element of the list is passed to `System.out.println`.

   Equivalent Lambda:

   ```java
   list.forEach(s -> System.out.println(s));
   ```

---

4. **Reference to a constructor**
   Syntax: `ClassName::new`
   Example:

   ```java
   Supplier<List<String>> listSupplier = ArrayList::new;
   List<String> myList = listSupplier.get();
   myList.add("Java 8");
   System.out.println(myList); // [Java 8]
   ```

   Equivalent Lambda:

   ```java
   Supplier<List<String>> listSupplier = () -> new ArrayList<>();
   ```

---

### 🔹 When to Use Method References?

* When your lambda is **only calling an existing method**.
* To make the code more **readable and concise**.

---

### 🔹 Practical Examples

1. Sorting with Comparator:

   ```java
   List<String> list = Arrays.asList("Banana", "Apple", "Mango");
   Collections.sort(list, String::compareToIgnoreCase);
   System.out.println(list); // [Apple, Banana, Mango]
   ```

2. Printing elements:

   ```java
   list.forEach(System.out::println);
   ```

3. Constructor reference in Streams:

   ```java
   Stream<String> names = Stream.of("A", "B", "C");
   List<StringBuilder> builders = names.map(StringBuilder::new).toList();
   ```

---

### 🔹 Benefits of Method References

1. Cleaner and shorter than lambdas.
2. Improves readability of code.
3. Works seamlessly with functional interfaces.
4. Reduces boilerplate when the lambda body just calls a method.

---

### 🔹 Limitations

1. Only applicable when lambda **just calls an existing method**.
2. If logic is more than a single method call, you must use a full lambda.

---

✨ In summary:
**Method & Constructor References** are just syntactic sugar for lambda expressions when you’re calling existing methods/constructors. They make the code elegant, concise, and more readable.


## 🔹 Common Syntax – `ClassName::methodName`

Java 8 ne deliberately **ek hi syntax** rakha:

```java
ClassName::methodName
```

Isse readability badhti hai aur lambda ke saath easily replace ho jata hai.
Lekin **binding (static vs instance)** runtime par context se clear hoti hai.

---

## 🔹 1. Static Method Reference

* Syntax: `ClassName::staticMethod`
* Example:

  ```java
  Function<String, Integer> f = Integer::parseInt;
  System.out.println(f.apply("123"));  // 123
  ```
* Yaha `Integer.parseInt(String)` static hai, isliye object ki zaroorat nahi.

---

## 🔹 2. Instance Method Reference (Bound/Unbound)

### (a) **Unbound** – ClassName::instanceMethod

* Syntax same lagta hai: `ClassName::instanceMethod`
* Lekin iska matlab hota hai: **pehla argument ban jaata hai target object**.
* Example:

  ```java
  Function<String, Integer> f = String::length;
  System.out.println(f.apply("hello"));  // 5
  ```

  Internally yeh lambda ke equal hai:

  ```java
  (String s) -> s.length()
  ```

### (b) **Bound** – object::instanceMethod

* Jab tumhe ek fixed object pe call karna hai:

  ```java
  String prefix = "Java";
  Predicate<String> p = prefix::startsWith;
  System.out.println(p.test("Java8"));  // true
  ```

  Yaha `prefix` already bound hai, extra parameter ki need nahi.

---

## 🔹 3. Constructor Reference

Bas syntax thoda alag lagta hai:

```java
Supplier<List<String>> s = ArrayList::new;
System.out.println(s.get().getClass()); // class java.util.ArrayList
```

---

## 🔹 Trick to Remember (Interview Note 📝)

1. **Static → ClassName::methodName** → direct map hota hai `(args) -> ClassName.methodName(args)`
2. **Unbound Instance → ClassName::methodName** → map hota hai `(obj, args) -> obj.methodName(args)`
3. **Bound Instance → obj::methodName** → map hota hai `(args) -> obj.methodName(args)`

👉 Matlab syntax same hai (`::`), **difference context se decide hota hai** ki yeh static hai ya instance.

---

📌 Short Note for Pen-Paper:

```
Static:   Class::staticMethod   → (args) -> Class.staticMethod(args)
Unbound:  Class::instanceMethod → (obj, args) -> obj.instanceMethod(args)
Bound:    obj::instanceMethod   → (args) -> obj.instanceMethod(args)
Ctor:     Class::new            → () -> new Class()
```

### When to use Construcotr Reference
## 🔹 1. Object Creation Simplification

Jab tumhe lambda me bar-bar `new` likhna padta hai, usko short karne ke liye constructor reference use karte ho.
Example:

```java
Supplier<List<String>> listSupplier = ArrayList::new;
// Equivalent: () -> new ArrayList<>()
```

Yaha readability better ho jaati hai aur boilerplate code kam hota hai.

---

## 🔹 2. Streams ke saath Collectors

Streams me `.collect()` ke andar constructor reference **bahut common** hai.
Example:

```java
List<String> names = Stream.of("a", "b", "c")
                           .collect(Collectors.toCollection(ArrayList::new));
```

Yaha `ArrayList::new` ek constructor reference hai jo Collector ko batata hai ki output kis collection me store karna hai.

---

## 🔹 3. Mapping Objects (Factory Style)

Jab tumhe ek list ke elements se naye objects banane hote hain:

```java
List<String> names = Arrays.asList("Ram", "Shyam", "Geeta");

List<Person> people = names.stream()
                           .map(Person::new)   // yaha Person(String name) constructor call hoga
                           .collect(Collectors.toList());
```

Yaha har `String` ke liye ek `Person` object create ho jaata hai.

---

## 🔹 4. Constructor Overloading with Functional Interfaces

Tum custom factory methods banane ke liye bhi use karte ho:

```java
interface PersonFactory {
    Person create(String name, int age);
}

PersonFactory factory = Person::new; // matches Person(String, int)
Person p = factory.create("Danish", 28);
```

Yaha compiler automatically constructor ko functional interface ke signature se map kar deta hai.

---

## 🔹 Interview / Notes Point 📝

* **Supplier** → no-arg constructor reference (`ArrayList::new`)
* **Function** → 1-arg constructor (`Person::new` from `String`)
* **BiFunction** → 2-arg constructor (`Person::new` from `String, int`)
* Functional interface ke signature = jis constructor se match karega, wahi bind ho jaata hai.

---

👉 Short Formula for Notes:

```
ClassName::new
Used when: instead of lambda (args) -> new ClassName(args)
Works with: Supplier, Function, BiFunction, etc.
Mostly used in: Streams (collect, map), factory patterns, DI frameworks.
```

<mark>Demo Program to cover all above 4 ways</mark>

```java
import java.util.*;
import java.util.function.*;
import java.util.stream.*;

class Person {
    private String name;
    private int age;

    // Constructor 1 - no args
    public Person() {
        this.name = "Unknown";
        this.age = 0;
    }

    // Constructor 2 - 1 arg
    public Person(String name) {
        this.name = name;
        this.age = 0;
    }

    // Constructor 3 - 2 args
    public Person(String name, int age) {
        this.name = name;
        this.age = age;
    }

    @Override
    public String toString() {
        return name + " (" + age + ")";
    }
}

public class ConstructorReferenceDemo {
    public static void main(String[] args) {

        // 1. Object creation simplification (Supplier)
        Supplier<List<String>> listSupplier = ArrayList::new;
        List<String> fruits = listSupplier.get();
        fruits.add("Apple");
        fruits.add("Mango");
        System.out.println("Fruits: " + fruits);

        // 2. Streams + Collectors (toCollection)
        List<String> names = Stream.of("Ram", "Shyam", "Geeta")
                                   .collect(Collectors.toCollection(ArrayList::new));
        System.out.println("Names: " + names);

        // 3. Mapping objects (map with constructor reference)
        List<Person> people = names.stream()
                                   .map(Person::new) // calls Person(String name)
                                   .collect(Collectors.toList());
        System.out.println("People: " + people);

        // 4. Constructor overloading with custom factory
        BiFunction<String, Integer, Person> personFactory = Person::new;
        Person p = personFactory.apply("Danish", 28);
        System.out.println("Factory Person: " + p);
    }
}
```

## **Java 8 Main Functional Interfaces – Practical Notes**

### **1. Supplier<T>**

* **Kya karta hai?**
  Koi input nahi leta, sirf value return karta hai.
  "Value ka supplier" samajh le.
* **Kab use hota hai?**
  Jab value **on-demand generate** karni ho, e.g. DB se record lana, random number, current date, config value.
* **Signature:**

  ```java
  @FunctionalInterface
  public interface Supplier<T> {
      T get();
  }
  ```
* **Example:**

  ```java
  Supplier<Double> randomSupplier = () -> Math.random();
  System.out.println(randomSupplier.get()); // e.g. 0.5471
  ```

---

### **2. Consumer<T>**

* **Kya karta hai?**
  Ek input leta hai, kuch return nahi karta.
  "Value consume karke kaam karna" — jaise print karna, DB me save karna, log karna.
* **Kab use hota hai?**
  Jab tumhe sirf input par action lena ho (output kaam ka nahi).
* **Signature:**

  ```java
  @FunctionalInterface
  public interface Consumer<T> {
      void accept(T t);
  }
  ```
* **Example:**

  ```java
  Consumer<String> printConsumer = s -> System.out.println(s);
  printConsumer.accept("Hello World"); // prints Hello World
  ```

---

### **3. Predicate<T>**

* **Kya karta hai?**
  Ek input leta hai, boolean return karta hai.
  "Condition check" ke liye.
* **Kab use hota hai?**
  Filter operations, validation logic.
* **Example:**

  ```java
  Predicate<Integer> isEven = n -> n % 2 == 0;
  System.out.println(isEven.test(4)); // true
  ```

---

### **4. Function<T, R>**

* **Kya karta hai?**
  Ek input leta hai, uska transformed output return karta hai.
* **Kab use hota hai?**
  Mapping, data transformation.
* **Example:**

  ```java
  Function<String, Integer> lengthFunc = str -> str.length();
  System.out.println(lengthFunc.apply("Danish")); // 6
  ```

---

### **5. Difference – Short Table**

| Interface | Input | Output      | Purpose                         |
| --------- | ----- | ----------- | ------------------------------- |
| Supplier  | ❌     | ✅           | Value supply karna (no input)   |
| Consumer  | ✅     | ❌           | Value consume karna (no return) |
| Predicate | ✅     | ✅(boolean)  | Condition check karna           |
| Function  | ✅     | ✅(any type) | Input → Output transform        |

---

💡 **Easy Memory Trick:**

* **Supplier** = "Give me something" (no input, only output)
* **Consumer** = "I’ll take it and do something" (input, no output)
* **Predicate** = "Is it true?" (boolean output)
* **Function** = "Convert this into that" (input → output)

* **Supplier** = "Bhai, mujhe kuch chahiye" → Tumse koi input nahi lega, sirf dega.
* **Consumer** = "Bhai, tu lele aur kaam kar" → Tum se input lega, return nahi karega.

---

## **Supplier**

* **Main kaam:** **"Mujhe kuch naya de"** → No input, only output.
* **Random value** banana ek common use-case hai (Math.random(), UUID, OTP, etc.)
* **Lekin sirf random tak limited nahi hai** — koi bhi value jo tum run-time pe banate ho, chahe woh DB se ho, config file se, ya default value ke form me ho.
* **Example list:**

  1. Random number generate karna
  2. Current date/time dena
  3. DB se ek record fetch karna (without any external input)
  4. Fallback/default value dena in `Optional.orElseGet()`

---

## **Consumer**

* **Main kaam:** **"Yeh le, ispe kaam kar"** → Takes input, does action, no return.
* **Printing** ek common use-case hai (System.out.println).
* **Lekin print tak limited nahi hai** — koi bhi operation jo tum input par karte ho bina kuch return kiye.
* **Example list:**

  1. Console me print karna (`System.out.println`)
  2. List ke elements iterate karke kaam karna (`forEach`)
  3. Logging
  4. File me write karna
  5. DB me save karna

---

💡 **Memory Hook:**

* Supplier → "Give me" (Generate/Fetch)
* Consumer → "Take it" (Process/Use)

💡 **Key Takeaways**

* **Supplier** = “Value banani/fetch karni hai” → Mostly **get()** method call hota hai.
* **Consumer** = “Value pe kaam karna hai” → Mostly **accept()** method call hota hai ya **forEach()** ke andar use hota hai.

# Comparator
Chalo bhai, **Comparator** ka concept ekdum clear karte hain with examples, Java 8 ke angle se.

---

## **1. Comparator Kya Hai?**

* **Definition:** Comparator ek functional interface hai (`java.util.Comparator<T>`) jo define karta hai ki do objects ko kaise compare karna hai.
* **Purpose:** Custom sorting rules banane ke liye.
* **Method:**

  * Java 7 tak → `compare(T o1, T o2)` implement karte the.
  * Java 8 me → Lambdas, method references, aur helper methods (`comparing`, `thenComparing`, etc.) ka use.

---

## **2. Basic Example – Single Field Sort**

```java
List<String> names = Arrays.asList("John", "Alice", "Bob");

names.sort((a, b) -> a.compareTo(b)); // ascending
System.out.println(names); // [Alice, Bob, John]
```

---

## **3. Java 8 Style – Method References**

```java
names.sort(Comparator.naturalOrder()); // ascending
names.sort(Comparator.reverseOrder()); // descending
```

---

## **4. Object List Sort – Single Field**

```java
List<Employee> list = Arrays.asList(
    new Employee("John", 30),
    new Employee("Alice", 25)
);

list.sort(Comparator.comparing(Employee::getAge));
```

* `comparing()` ek key extractor leta hai (yaha `getAge`).

---

## **5. Multi-Level Sort (thenComparing)**

```java
list.sort(Comparator.comparing(Employee::getDept)
                    .thenComparing(Employee::getSalary, Comparator.reverseOrder()));
```

* Pehle department ke hisaab se sort, phir salary ke hisaab se (descending).

---

## **6. Null-Safe Sorting**

```java
list.sort(Comparator.comparing(Employee::getName,
        Comparator.nullsLast(String::compareTo)));
```

* Null values ko last me bhej diya.

---

## **7. Reverse Order Easily**

```java
list.sort(Comparator.comparing(Employee::getAge).reversed());
```

---

## **8. Complex Example – Streams**

```java
employees.stream()
         .sorted(Comparator.comparing(Employee::getDept)
                            .thenComparing(Employee::getName))
         .forEach(System.out::println);
```

## 🔑 Comparator Tricky Bullet Points

* **naturalOrder() / reverseOrder()**
  → Only work when elements are *Comparable* (like String, Integer).
  → Useful when you don’t want custom logic.

* **comparing(...) methods**
  → *comparing(Function)* = choose property for sorting.
  → *comparingInt/Long/Double* = primitive properties (faster, no boxing).
  → *comparing with Comparator* = nested sorting logic.

* **thenComparing(...) methods**
  → Secondary / tie-breaker sorting.
  → Chaining works: sort by name → then by age → then by salary.

* **nullsFirst / nullsLast**
  → Handle `null` values gracefully in sorting.
  → Must be wrapped with an existing comparator.

* **reversed()**
  → Flip comparator direction.
  → Often used after `comparing(...)`.

* **Map.Entry.comparingByKey / comparingByValue**
  → Directly sort a `Map` by key or value.
  → Overloaded versions accept custom comparator for keys/values.

---

💡 **Key Points to Remember**

* **`comparing()`** – ek field ke basis pe sort.
* **`thenComparing()`** – secondary, tertiary sorting rules.
* **`reversed()`** – reverse order banane ke liye.
* **`nullsFirst()` / `nullsLast()`** – null handle karne ke liye.
* **Java 8 Lambdas** – boilerplate code kam.

---

`naturalOrder()` aur `reverseOrder()` dono **Comparator interface ke predefined static methods** hain jo Java 8 me introduce hue:

## **1. Comparator.naturalOrder()**

* **Definition:**
  Static method jo ek comparator return karta hai jo **Comparable** objects ko **ascending** (natural) order me sort karta hai.
* **Kaam karta hai:**

  * Strings → A to Z
  * Numbers → smallest to largest
  * Dates → oldest to newest
* **Example:**

```java
List<Integer> nums = Arrays.asList(5, 1, 3);
nums.sort(Comparator.naturalOrder());
System.out.println(nums); // [1, 3, 5]
```

---

## **2. Comparator.reverseOrder()**

* **Definition:**
  Static method jo ek comparator return karta hai jo **Comparable** objects ko **descending** order me sort karta hai.
* **Kaam karta hai:**

  * Strings → Z to A
  * Numbers → largest to smallest
  * Dates → newest to oldest
* **Example:**

```java
List<Integer> nums = Arrays.asList(5, 1, 3);
nums.sort(Comparator.reverseOrder());
System.out.println(nums); // [5, 3, 1]
```

---

### 1) Map ko **value** ke basis par sort karke LinkedHashMap banao

**Question:** `Map<String,Integer> scores` ko **value desc** order me sort karke insertion-ordered map chahiye.
**Solution:**

```java
Map<String,Integer> sorted =
    scores.entrySet().stream()
          .sorted(Map.Entry.<String,Integer>comparingByValue().reversed())
          .collect(Collectors.toMap(
              Map.Entry::getKey, Map.Entry::getValue,
              (a,b) -> a, LinkedHashMap::new));
```

*Note: `comparingByValue().reversed()` + `LinkedHashMap`.*

```java
.collect(Collectors.toMap(
    Map.Entry::getKey, 
    Map.Entry::getValue,
    (a, b) -> a, 
    LinkedHashMap::new));
```

### Step-by-step explanation

1. **`Collectors.toMap(...)`**  
   This is a collector used in Java Streams to accumulate elements into a `Map`.

2. **`Map.Entry::getKey`**  
   - This is the **key mapper function**.  
   - For each element in the stream (which is a `Map.Entry`), it extracts the **key** using `getKey()`.

3. **`Map.Entry::getValue`**  
   - This is the **value mapper function**.  
   - For each element in the stream, it extracts the **value** using `getValue()`.

   👉 Together, these two lines define how keys and values are set in the resulting map.

4. **`(a, b) -> a`**  
   - This is the **merge function**.  
   - It tells the collector what to do if there are duplicate keys.  
   - In this case, if two entries have the same key, it keeps the **first value (`a`)** and ignores the second (`b`).  
   - Example: If you had entries `(k, v1)` and `(k, v2)`, the resulting map will store `(k, v1)`.

5. **`LinkedHashMap::new`**  
   - This specifies the **map supplier**.  
   - Instead of using a regular `HashMap`, it uses a `LinkedHashMap`.  
   - The difference is that `LinkedHashMap` preserves **insertion order**, so the final map will keep the order in which elements were processed.

---

### Putting it all together
- The stream produces `Map.Entry` objects.  
- For each entry:
  - The key is set using `getKey()`.
  - The value is set using `getValue()`.
- If duplicate keys appear, the first value is kept (`(a, b) -> a`).  
- The final result is stored in a `LinkedHashMap`, preserving order.
---

<mark>Collectors.toMap()</mark>

Q: so how many variable we can pass in Collectors.toMap()?
 `Collectors.toMap()` actually has **four possible parameters** you can pass, depending on which overload you use. Let’s break them down:
---
### Overloads of `Collectors.toMap`

1. **Two-argument version**  
   ```java
   Collectors.toMap(keyMapper, valueMapper)
   ```
   - `keyMapper`: function to extract the key  
   - `valueMapper`: function to extract the value  
   - Uses a default merge function (throws an exception if duplicate keys are found).  
   - Backed by a `HashMap`.

---

2. **Three-argument version**  
   ```java
   Collectors.toMap(keyMapper, valueMapper, mergeFunction)
   ```
   - Adds a **merge function** to handle duplicate keys.  
   - Still backed by a `HashMap`.

---

3. **Four-argument version**  
   ```java
   Collectors.toMap(keyMapper, valueMapper, mergeFunction, mapSupplier)
   ```
   - Adds a **map supplier** to control the type of map created (e.g., `LinkedHashMap::new`, `TreeMap::new`).  
   - This is the version you’re using in your code.

---

### So, how many variables can you pass?
- **Minimum:** 2 (keyMapper, valueMapper)  
- **Maximum:** 4 (keyMapper, valueMapper, mergeFunction, mapSupplier)

---

### Quick Example

```java
// 2 args
.collect(Collectors.toMap(
    Map.Entry::getKey,
    Map.Entry::getValue));

// 3 args
.collect(Collectors.toMap(
    Map.Entry::getKey,
    Map.Entry::getValue,
    (a, b) -> a));

// 4 args
.collect(Collectors.toMap(
    Map.Entry::getKey,
    Map.Entry::getValue,
    (a, b) -> a,
    LinkedHashMap::new));
```

### 3) **Null-safe** multi-field sort: name asc, joinDate (nullable) asc with nulls last

**Solution:**

```java
    Comparator.comparing(Employee::getName, String::compareToIgnoreCase)
```

### 4) **Custom Enum order** (business priority)

### 7) **Version strings** sort: "1.9.2", "1.10.0", "2.0" (lexicographic se galat)

**Question:** Semantic version order chahiye.
**Solution:**

```java
Comparator<String> verCmp = (v1, v2) -> {
    int[] a = Arrays.stream(v1.split("\\.")).mapToInt(Integer::parseInt).toArray();
    int[] b = Arrays.stream(v2.split("\\.")).mapToInt(Integer::parseInt).toArray();
    int n = Math.max(a.length, b.length);
    for (int i = 0; i < n; i++) {
        int x = i < a.length ? a[i] : 0;
        int y = i < b.length ? b[i] : 0;
        if (x != y) return Integer.compare(x, y);
    }
    return 0;
};
versions.sort(verCmp);
```

*Note: Missing parts ko 0 treat karo.*

# 11) “Version string” comparator (semantic-like)

```java
Comparator<String> versionCmp = (v1, v2) -> {
    int[] a = Arrays.stream(v1.split("\\.")).mapToInt(Integer::parseInt).toArray();
    int[] b = Arrays.stream(v2.split("\\.")).mapToInt(Integer::parseInt).toArray();
    int n = Math.max(a.length, b.length);
    for (int i = 0; i < n; i++) {
        int x = i < a.length ? a[i] : 0;
        int y = i < b.length ? b[i] : 0;
        if (x != y) return Integer.compare(x, y);
    }
    return 0;
};

var versions = new ArrayList<>(List.of("1.9.2", "1.10.0", "2.0", "1.9"));
versions.sort(versionCmp);      // [1.9, 1.9.2, 1.10.0, 2.0]
```

### 🔹 Sorting Without Comparator → Internally Comparable Use Hota Hai

Haan, jab bhi tum **Comparator explicitly pass nahi karte**, Java internally check karta hai ki element class ne `Comparable<T>` implement kiya hai ya nahi.

### 🔹 Example 1: Works Fine (because Comparable implemented)


### 🔹 Example 2: Fails (no Comparable implemented)

```java
class Student {

}

        List<Student> list = Arrays.asList(new Student(2), new Student(1));
        Collections.sort(list); // ❌ Runtime error
    }
```

```java
class Student implements Comparable<Student> {
    int id;
    Student(int id) { this.id = id; }

    @Override
    public int compareTo(Student s) {
        return this.id - s.id;  // natural order by id
    }

    @Override
    public String toString() {
        return "Student(" + id + ")";
    }
}

public class Test {
    public static void main(String[] args) {
        List<Student> list = Arrays.asList(new Student(2), new Student(1));
        Collections.sort(list);  // uses compareTo()
        System.out.println(list); // [Student(1), Student(2)]
    }
}
```

---

## 🔹 `Collections.reverseOrder()`

* Ye **utility method** hai jo `Collections` class (java.util) me defined hai.
* Do overloads:

  1. `Collections.reverseOrder()` → returns a comparator that reverses the **natural ordering**.
  2. `Collections.reverseOrder(Comparator cmp)` → returns a comparator that reverses the **given comparator**.
* Basically, ek “factory method” hai jo comparator banake deta hai.

**Example:**

```java
List<Integer> list = Arrays.asList(5, 1, 3);
Collections.sort(list, Collections.reverseOrder()); // natural reverse
```

---

## 🔹 `Comparator.reverseOrder()`

* Ye **static method** hai `Comparator` interface me (Java 8 se).
* Ye bhi ek comparator return karta hai jo natural ordering ko reverse karega.
* Equivalent hai `Collections.reverseOrder()` ke bina parameter wale version ke.
* Iske paas overloaded version nahi hai (sirf natural order reverse karta hai).

**Example:**

```java
list.sort(Comparator.reverseOrder());
```

---

## 🔑 Difference in Simple Words

| Aspect      | `Collections.reverseOrder()`     | `Comparator.reverseOrder()`            |
| ----------- | -------------------------------- | -------------------------------------- |
| Location    | `java.util.Collections` class    | `java.util.Comparator` interface       |
| Overloads   | Yes (no-arg + takes Comparator)  | No (only natural reverse)              |
| Since       | Java 2 se (bahut purana)         | Java 8 se (naya, modern style)         |
| Usage       | Mostly with `Collections.sort()` | Mostly with `List.sort()` / Stream API |
| Flexibility | Can reverse custom comparator    | Only reverses natural order            |

---

## ✅ Conclusion

* Agar tumhe **natural order reverse** karna hai → dono chalenge (`Collections.reverseOrder()` ya `Comparator.reverseOrder()`).
* Agar tumhe **custom comparator ko reverse** karna hai → sirf `Collections.reverseOrder(customComparator)` kaam aayega.
* Practical coding me Java 8+ ke saath log zyada `Comparator.reverseOrder()` + `List.sort()`/`Stream.sorted()` use karte hain.

### 🔹 Pehle (Java 7 and below)

* Interfaces (`Comparator`, `Function`, `Predicate`, `Collectors`) khud ke static methods ke saath aane lage.
* Isse:

  * Code concise hua (lambda + method references).
  * Utility class ke imports kam ho gaye.
  * API ekdum “fluent” style ho gayi.

**Example:**

# 📘 Java Sorting – Master Notes

---

## 1. Sorting Basics

* **Sorting** = arranging objects in a particular order (ascending, descending, custom).
* Java me sorting ke liye `Comparable` aur `Comparator` dono concepts use hote hain.

---

## 2. Comparable Interface

* Package: `java.lang`
* **Natural Ordering** provide karta hai.
* Ek class ko **self-sorting** banata hai.
* **Method:**

  ```java
  public int compareTo(T o);
  ```
* Returns:

  * `negative` → this < o
  * `0` → this == o
  * `positive` → this > o

### ✅ Examples

* String, Wrapper classes, BigInteger, BigDecimal already implement `Comparable`.
* Custom class:

  ```java
  class Student implements Comparable<Student> {
      int age;
      public int compareTo(Student s) {
          return this.age - s.age;
      }
  }
  ```
* Use:

  ```java
  Collections.sort(list); // uses compareTo()
  ```

---

## 3. Comparator Interface

* Package: `java.util`
* **External sorting logic** define karta hai (object ke andar compareTo nahi chipkana padta).
* **Method:**

  ```java
  public int compare(T o1, T o2);
  ```

### ✅ Examples

```java
Collections.sort(list, new Comparator<Student>() {
    public int compare(Student s1, Student s2) {
        return s1.name.compareTo(s2.name);
    }
});
```

---

## 4. Difference – Comparable vs Comparator

| Feature        | Comparable                  | Comparator                         |
| -------------- | --------------------------- | ---------------------------------- |
| Package        | java.lang                   | java.util                          |
| Method         | compareTo                   | compare                            |
| Used For       | Natural order               | Custom order                       |
| Modifies Class | Yes (implements Comparable) | No, external class or lambda       |
| Sort Call      | Collections.sort(list)      | Collections.sort(list, comparator) |

---

## 5. Java 8 Enhancements in Comparator

Java 8 ne Comparator ko **functional interface** banaya aur static/default methods add kiye:

* `comparing(Function)`
* `thenComparing(Function)`
* `reversed()`
* `naturalOrder()`
* `reverseOrder()`
* `nullsFirst()`, `nullsLast()`

### ✅ Examples

```java
list.sort(Comparator.comparing(Student::getAge));
list.sort(Comparator.comparing(Student::getAge).reversed());
list.sort(Comparator.comparing(Student::getAge)
                    .thenComparing(Student::getName));
```

---

## 6. Sorting Utilities

* **Collections.sort(list)** → works with Comparable or Comparator.
* **Collections.sort(list, comparator)** → custom order.
* **List.sort(comparator)** → Java 8 preferred style.

1. `Collections.reverseOrder()`
2. `Comparator.reverseOrder()`
3. `Comparator.comparing(...).reversed()`
4. `Collections.reverse(list)` (in-place, not comparator-based)
5. `stream()..sorted((a, b) -> b.compareTo(a))`

3. **BigInteger** and **BigDecimal** bhi Comparable hai (precision ke liye).
4. **Wrapper classes + String** sabhi Comparable hai.

<mark>NOTE For Sorting</mark>

4. Sort/Sorted agar method parameter me kuch pass nhi kiya to Comparable ka use hai natural sorting
5. Sort/Sorted agar method parameter me Comparator ki static methods pass for by Comparator.
6. Sort/Sorted agar method parameter me Comparator ka logic (a,b)->a.length()-b.length() and (a, b) -> a.compareTo(b) pass for by Comparator.

# Collect Method

* **Stream ka terminal operation** hai → matlab stream ko end karta hai aur result banata hai.
* Input: `Collector` (ek predefined ya custom strategy).
* Output: koi **collection**, **map**, **string**, ya **numeric summary** ban sakta hai.

---

## 2. Syntax

```java
<R, A> R collect(Collector<? super T, A, R> collector)
```

* `T`: Stream elements ka type
* `A`: Accumulator (intermediate storage)
* `R`: Final result type

Matlab `.collect()` me **Collector object pass hota hai** jo batata hai result kaise banana hai.

**Collector<? super T, A, R>**
Java allows collectors that accept supertypes of T.
This is useful because many collectors are written in terms of more general types.
```java
Stream<String> s = Stream.of("a", "b", "c");
// Collector works with Object, not just String
Collector<Object, ?, List<Object>> c = Collectors.toList();
List<Object> result = s.collect(c); // ✅ works because of ? super T
```
If the method required Collector<T, A, R>, this would fail — you couldn’t use a collector that expects Object with a Stream<String>.
try to understand the difference between the final output (string vs object)  
Below Ex1 Because the collector is Collector<Object, ?, List<Object>>, the wildcard ? super T allows String elements to be treated as Object.
```java
1. Object 
Stream<String> s = Stream.of("a", "b", "c");
Collector<Object, ?, List<Object>> c = Collectors.toList();
List<Object> result = s.collect(c); // works because of ? super T

2. String
Stream<String> s = Stream.of("a", "b", "c");
```

## 2. 3-arg `collect()`

Syntax:

```java
<R, A> R collect(Supplier<R> supplier,
                 BiConsumer<R, ? super T> accumulator,
                 BiConsumer<R, R> combiner)
```

* **Supplier** → result container banata hai (jaise `ArrayList::new`).
* **Accumulator** → har element ko result me add karta hai.
* **Combiner** → parallel stream me alag alag partial results ko merge karta hai.

👉 Example:

```java
List<String> names = List.of("A", "BB", "CCC");

List<String> result = names.stream()
        .collect(ArrayList::new,          // Supplier
                 List::add,               // Accumulator
                 List::addAll);           // Combiner

System.out.println(result); // [A, BB, CCC]
```

* Yahi kaam `Collectors.toList()` karta hai internally.
* Tum ise tab use karte ho jab tumhe **custom accumulation logic** likhni hai without `Collectors`.

## 3. `Collector.of()`

Ye tum apna **custom Collector** define karne ke liye use karte ho.

Syntax:

```java
Collector.of(Supplier, Accumulator, Combiner, Finisher)
```

* `Supplier` → result container banata hai.
* `Accumulator` → elements ko result me add karta hai.
* `Combiner` → parallel processing ke liye results merge karta hai.
* `Finisher` (optional) → final transformation (e.g. convert `StringBuilder` → `String`).

👉 Example: Stream ko **comma separated string** banane ka custom collector:

```java
List<String> names = List.of("A", "BB", "CCC");

Collector<String, StringBuilder, String> joiningWithBrackets =
        Collector.of(
                StringBuilder::new,               // Supplier
                (sb, s) -> sb.append("[").append(s).append("]"),  // Accumulator
                StringBuilder::append,            // Combiner
                StringBuilder::toString           // Finisher
        );

String result = names.stream().collect(joiningWithBrackets);

System.out.println(result); // [A][BB][CCC]
```

* Ye tum `Collectors.joining(" ")` se bhi kar sakte ho, but `Collector.of()` tumhe **poora control** deta hai.

Great question! Short, crisp answer:

## Combiner kab use hota hai?

* **Primary use:** **Parallel streams** me. Jab stream ko parallel run kiya jata hai, multiple threads alag-alag partial containers banate hain; **combiner** un partial results ko **merge** karta hai.
* **Sequential stream** me: **usually combiner call nahi hota.** Ek hi container me saara accumulation hota hai, isliye merge ki zarurat nahi padti.

## Important nuance (interview-worthy)

* **Contract ke hisaab se** combiner **hamesha sahi likhna chahiye**, chahe aaj sequential ho. Kyunki:

  * Aap kal `parallel()` laga do to wahi collector parallel ho jayega.
  * Stream implementation theoretically sequential me bhi combiner use kar sakti hai (rare), to **combiner associative & correct** hona chahiye.

## 3-arg `collect(...)` vs `Collector.of(...)`

* **3-arg `collect(supplier, accumulator, combiner)`**

  * Sequential: combiner **normally** call nahi hota.
  * Parallel: combiner **definitely** partial containers merge karega.
* **`Collector.of(...)`** (prebuilt/custom collectors ka pattern)

  * Same rule: combiner ka role **mostly** parallel me hota hai, lekin aapko **always correct** likhna hota hai (associative merge).

## Thumb rules for notes

* “**Combiner = merge two containers**”
* “**Mostly parallel me needed**; sequential me usually unused.”
* “**Always correct & associative** likho, taaki parallel me safe rahe.”
* Typical pattern lists/sets ke liye:

  * **List:** `(l1, l2) -> { l1.addAll(l2); return l1; }`
  * **Set:**  `(s1, s2) -> { s1.addAll(s2); return s1; }`
  * **Map (frequency):** `(m1, m2) -> { m2.forEach((k,v)-> m1.merge(k,v,Integer::sum)); return m1; }`

## Direct Answer to Your Question

➡️ Nahi bhai 😅

# Sequential vs Parallel — `collect(supplier, accumulator, combiner)`

| Phase             | Sequential Stream (`stream()`)                                         | Parallel Stream (`parallelStream()`)                                  |
| ----------------- | ---------------------------------------------------------------------- | --------------------------------------------------------------------- |
| **Supplier**      | 1 hi container banta hai (e.g., ek `ArrayList`)                        | Har worker thread apna **partial** container banata hai               |
| **Accumulator**   | **Har element** ke liye call hota hai (isi se container fill hota hai) | **Har element** per-thread container me add hota hai                  |
| **Combiner**      | **Usually NOT called** (merge ki zarurat nahi)                         | **必 called** to merge partial containers (`addAll`, `putAll`, custom) |
| **Order**         | Source order rakhna easy (agar stream ordered hai)                     | Order preserve karne ke liye extra cost (use `.forEachOrdered`)       |
| **Thread-safety** | Needed nahi (single thread)                                            | **Container ya combining logic thread-safe/associative honi chahiye** |
| **Performance**   | Simple, predictable                                                    | Data size & cores pe depend—sahi combiner likhna zaroori              |

## Thumb rules (pen-paper):

* **Accumulator always runs** (seq & parallel).
* **Combiner mostly parallel me** (merge partial results).
* Combiner **associative** likho: `combine(a,b)` + `combine(b,c)` order-independent ho.
* Common patterns:

  * List: `(l1,l2) -> { l1.addAll(l2); return l1; }`
  * Set: `(s1,s2) -> { s1.addAll(s2); return s1; }`
  * Map (freq): `(m1,m2) -> { m2.forEach((k,v)-> m1.merge(k,v,Integer::sum)); return m1; }`

## stream.map() vs Collectors.mapping()
---

<mark>NOTE:</mark>  
>1. stream.map() sare elements pe lagega
>2. Collectors.mapping() keval group of elements pe lagega using groupingBy/partitioningBy()==Downstream Collectors like mapping/filtering/summingInt etc

## 1. `stream().map()`

* **Stage**: *Intermediate operation* (stream pipeline ke beech me).
* **Scope**: Har element ko transform karke ek naya stream banata hai.
* **Example**:

  ```java
  List<String> names = List.of("Danish", "Khan");

  List<Integer> lengths = names.stream()
                               .map(String::length)   // map here
                               .collect(Collectors.toList());
  ```
* **Kaam**: Pure pipeline pe effect — jitna aage ka stream hoga, sab transform hua data dekhega.

---

## 2. `Collectors.mapping()`

* **Stage**: *Collector ke andar transformer* (aggregation ke waqt).
* **Scope**: Sirf aggregation ke liye incoming values transform karta hai, pure stream pipeline nahi badalta.
* **Example**:

  ```java
  Map<Integer, List<String>> grouped = names.stream()
      .collect(Collectors.groupingBy(
          String::length,
          Collectors.mapping(String::toUpperCase, Collectors.toList()) // mapping here
      ));
  ```
* **Kaam**: Jab `groupingBy`/`partitioningBy` jaise collectors use karte ho, tab direct aggregation ke andar hi transformation ho jata hai.

---

## 🔑 **Main Difference**

| Feature            | `map()` (Stream)               | `Collectors.mapping()` (Collector)    |
| ------------------ | ------------------------------ | ------------------------------------- |
| **Where used?**    | Before terminal op (`collect`) | Inside terminal collector (`collect`) |
| **Effect**         | Transforms entire stream       | Transforms only *collected subgroup*  |
| **Pipeline scope** | Affects all downstream ops     | Limited to one collector branch       |
| **Use case**       | General-purpose transformation | Aggregation-specific transformation   |

---

## 🔍 Why do we need both?

1. **map()** = Universal transformer → har element pe apply hota hai, aur pure pipeline ko badalta hai.
2. **mapping()** = Local transformer → sirf collector ke context me chahiye hota hai.

👉 Agar `mapping()` na hota, to groupingBy/partitioningBy likhte time pehle alag stream map karna padta, phir collect karna padta. Ye extra pass aur boilerplate badha deta.

---

## ⚡ Quick Example Difference

Without `Collectors.mapping()`:

```java
Map<Integer, List<String>> grouped = names.stream()
    .map(String::toUpperCase)   // step outside
    .collect(Collectors.groupingBy(String::length));
```

With `Collectors.mapping()`:

```java
Map<Integer, List<String>> grouped = names.stream()
    .collect(Collectors.groupingBy(
        String::length,

    ));
```

👉 Dono same kaam karte hain, but `Collectors.mapping()` **directly fit hota hai aggregation context** me, aur readability/maintainability improve karta hai (especially nested collectors me).

* **Before grouping (pipeline `filter`)** → sirf matching elements se **groups banenge**.
* **Inside grouping (downstream `Collectors.filtering`)** → group **banega classifier se**, par list ke andar sirf matching items jayenge (kabhi-kabhi empty lists bhi mil sakti hain).

```java
  people.stream()

        .collect(Collectors.groupingBy(Person::getCity, Collectors.counting()));

// but count only seniors within each city
// (filtering inside collector): All cities appear, even those with no seniors (count = 0).
Map<String, Long> citySeniorCount =
  people.stream()
        .collect(Collectors.groupingBy(
          Person::getCity,
          Collectors.filtering(p -> p.getAge() >= 60, Collectors.counting())
        ));
```

# DownStream Collector
In Java 8, a **downstream collector** is simply the *collector that’s applied to the elements inside a grouping, partitioning, or mapping operation*.

It’s called “downstream” because it sits **further down the data flow** inside a higher-level collector.

---

## **Where you see it**

When you use collectors like:

* `Collectors.groupingBy(...)`
* `Collectors.partitioningBy(...)`
* `Collectors.mapping(...)`

…you can pass **another collector** as an argument.
That other collector is the **downstream collector** — it defines **how to collect the values in each group/partition**.

---

## **Example**

### Without downstream collector

```java
Map<String, List<String>> result =
    people.stream()
          .collect(Collectors.groupingBy(Person::getCity));
```

Here, the default downstream collector is `Collectors.toList()` (Java adds it automatically).

---

### With downstream collector

```java
Map<String, Long> result =
    people.stream()
          .collect(Collectors.groupingBy(
              Person::getCity,              // classifier
              Collectors.counting()         // downstream collector
          ));
```

* `groupingBy` is the **upstream** collector.
* `counting()` is the **downstream** collector — it decides *how* to collect the values for each city (in this case, by counting).

---

## **Chaining downstream collectors**

You can nest them to transform data before collecting:

```java
Map<String, Set<String>> result =
    people.stream()
          .collect(Collectors.groupingBy(
              Person::getCity,
              Collectors.mapping(
                  Person::getName,         // map each person to their name
                  Collectors.toSet()       // downstream collector
              )
          ));
```

Here:

* `groupingBy` groups by city.
* `mapping` transforms each `Person` into just the name.
* `toSet()` collects those names into a set per group.

---

✅ **Key idea:**
A downstream collector processes the *values within each bucket* that the upstream collector has already divided. It’s “downstream” in the collection pipeline.

# Downstream Collectors — Quick Cheat Sheet (Java 8)

| Goal (per group)                       | Downstream collector                      | Result type                 | When to use                                     | Minimal snippet                                                               |
| -------------------------------------- | ----------------------------------------- | --------------------------- | ----------------------------------------------- | ----------------------------------------------------------------------------- |
| Count elements                         | `Collectors.counting()`                   | `Long`                      | You only need totals                            | `groupingBy(key, counting())`                                                 |
| Sum ints/longs/doubles                 | `summingInt/Long/Double(mapper)`          | `Integer/Long/Double`       | Numeric totals                                  | `groupingBy(key, summingInt(Foo::getQty))`                                    |
| Average                                | `averagingInt/Long/Double(mapper)`        | `Double`                    | Mean value                                      | `groupingBy(key, averagingDouble(Foo::getPrice))`                             |
| Min / Max by comparator                | `minBy(comp)` / `maxBy(comp)`             | `Optional<T>`               | Extremes                                        | `groupingBy(key, maxBy(comparing(Foo::getScore)))`                            |
| Full stats (count, sum, min, avg, max) | `summarizingInt/Long/Double(mapper)`      | `IntSummaryStatistics` etc. | Need many stats at once                         | `groupingBy(key, summarizingInt(Foo::getQty))`                                |
| Collect to list                        | `toList()`                                | `List<T>`                   | Keep order/duplicates                           | `groupingBy(key, toList())`                                                   |
| Collect to set                         | `toSet()`                                 | `Set<T>`                    | Unique values                                   | `groupingBy(key, toSet())`                                                    |
| Custom collection                      | `toCollection(Supplier)`                  | e.g., `LinkedList<T>`       | Specific type needed                            | `groupingBy(key, toCollection(LinkedList::new))`                              |
| Transform then collect                 | `mapping(mapper, downstream)`             | downstream’s type           | Map items before collecting                     | `groupingBy(key, mapping(Foo::getName, toSet()))`                             |
| Post-process result                    | `collectingAndThen(downstream, finisher)` | `U` (after finisher)        | Make result unmodifiable, unwrap Optional, etc. | `groupingBy(key, collectingAndThen(toList(), Collections::unmodifiableList))` |
| Reduce to one value                    | `reducing(identity, mapper, op)`          | type of identity            | Custom reductions                               | `groupingBy(key, reducing(0, Foo::getQty, Integer::sum))`                     |
| Concatenate strings                    | `joining(delim, prefix, suffix)`          | `String`                    | CSV/pretty output                               | `groupingBy(key, mapping(Foo::getName, joining(", ")))`                       |

> “Downstream” = the collector that runs **inside** `groupingBy/partitioningBy/mapping`, deciding how each bucket’s values get accumulated.

## Canonical Patterns

### 1) Count items per key

```java
Map<Category, Long> counts =
  items.stream().collect(groupingBy(Item::getCategory, counting()));
```

### 2) Sum per key

```java
Map<String, Integer> qtyByCity =
  orders.stream().collect(groupingBy(Order::getCity, summingInt(Order::getQuantity)));
```

### 3) Average per key

```java
Map<String, Double> avgPriceByBrand =
  phones.stream().collect(groupingBy(Phone::getBrand, averagingDouble(Phone::getPrice)));
```

### 4) Unique mapped values per key

```java
Map<String, Set<String>> namesByCity =
  people.stream().collect(groupingBy(Person::getCity,
    mapping(Person::getName, toSet())));
```

### 5) Max element per key

```java
Map<String, Optional<Employee>> topEarnerByDept =
  emps.stream().collect(groupingBy(Employee::getDept,
    maxBy(Comparator.comparing(Employee::getSalary))));
```

### 6) “Stats” per key (count/sum/min/avg/max at once)

```java
Map<String, IntSummaryStatistics> statsByDept =
  emps.stream().collect(groupingBy(Employee::getDept,
    summarizingInt(Employee::getAge)));
```

Use:

```java
IntSummaryStatistics s = statsByDept.get("Sales");
long count = s.getCount(); int min = s.getMin(); double avg = s.getAverage();
```

### 7) Post-process a bucket result

Make lists unmodifiable:

```java
Map<String, List<Order>> roListsByUser =
  orders.stream().collect(groupingBy(Order::getUser,
    collectingAndThen(toList(), Collections::unmodifiableList)));
```

Unwrap `Optional` with a default:

```java
Map<String, Employee> topOrDefault =
  emps.stream().collect(groupingBy(Employee::getDept,
    collectingAndThen(
      maxBy(Comparator.comparing(Employee::getSalary)),
      opt -> opt.orElse(Employee.DEFAULT)
    )));
```

### 8) Custom reduction per key

```java
Map<String, Integer> totalQty =
  lines.stream().collect(groupingBy(Line::getSku,
    reducing(0, Line::getQty, Integer::sum)));
```

### 9) Join strings per key

```java
Map<String, String> csvNamesByTeam =
  devs.stream().collect(groupingBy(Dev::getTeam,
    mapping(Dev::getName, joining(", "))));
```

### 10) Partition (boolean split) with downstream

```java
Map<Boolean, Long> adultsVsMinors =
  people.stream().collect(partitioningBy(p -> p.getAge() >= 18, counting()));
```

---

## Quick Tips (Java 8 focus)

* Prefer `mapping(..., toSet())` over collecting to list and then converting.
* Need min/max by **mapped** value? Use `Comparator.comparing(mapper)`.
* If you find yourself doing “collect, then transform”, try `collectingAndThen`.
* For two stats at once (e.g., sum + count), consider `summarizing*` then read fields.
* `filtering` and `flatMapping` are **Java 9+**, not in Java 8.


# Scenario of Stream Problems

## ✅ Scenario 1 – **Whole stream pe kaam** (global scope)

* Saare elements ek hi “bucket” me.
* Yaha hum use karte hain:

  * `map / filter / flatMap`
  * `mapToInt / sum() / average()`
  * `reduce(...)`
  * `collect(toList()/toSet()/joining())`

👉 Matlab tumhe **sab elements mila kar** transform/aggregate karna hai.

---

## ✅ Scenario 2 – **Groups/Partitions pe kaam** (per-key scope)

* Elements ko kisi key (department, city, category…) ke basis par todna.
* Yaha hum use karte hain:

  * `Collectors.groupingBy(...)`
  * `Collectors.partitioningBy(...)`
  * Aur unke andar downstream: `mapping`, `filtering`, `counting`, `summingInt`, `averagingDouble`, etc.

👉 Matlab tumhe **bucket-wise (sub-stream wise)** transformation/aggregation karna hai.

---

## 🔍 Are there more scenarios?

Bhai practically **major duniyaa isi do axis pe chalti hai**:

1. *Global pipeline transformations*
2. *Grouped/partitioned collectors*

Baaki jo cheezein hain (`peek`, `forEach`, `toArray`, `collectingAndThen`, `teeing` in Java 12) — ye sab **variants/shortcuts** hi hain inhi 2 ideas ke.

* `peek` → debugging ke liye side-effect in pipeline (not recommended for logic).
* `forEach` → side-effects, logging, DB calls (avoid heavy use).
* `collectingAndThen` → ek collector ke result pe final transformation (wrapper).
* `teeing` (Java 12) → ek hi pass me do collectors parallel run karna (e.g. min+max ek saath nikalna).

Par root me ye bhi **global ya per-group aggregation** ke hi flavors hain.

---

## 🔑 Final takeaway

Tumhare notes ke liye ek line likh lo:

👉 *Stream API ke operations do hi scenarios cover karte hain*

1. **Entire stream ko transform/aggregate karna** → pipeline ops (`map/filter/reduce/collect(toList)` etc.)
2. **Stream ko todna groups me aur per-group aggregation karna** → collectors (`groupingBy/partitioningBy` + downstream).

Baki sab (mapping, filtering, reducing, teeing, etc.) bas **syntactic sugar / helper variants** hain.

## 🔹 Custom Collector ka role

* **Collector.of(...)** ya **3-arg collect(supplier, accumulator, combiner)** tumhe poora control deta hai:

  * Kaise container banta hai (Supplier)
  * Kaise elements usme add hote hain (Accumulator)
  * Parallel stream me alag-alag containers ko kaise merge karna hai (Combiner)
  * (Optional) Final container ko kaise convert karna hai (Finisher)

## 🔹 Relationship with Scenario 1 & 2

### ✅ Scenario 1 (Global aggregation)

Normally tum likhte:

```java
int sum = numbers.stream().reduce(0, Integer::sum);
```

Custom collector alternative:

```java
int sum = numbers.stream()
    .collect(
        () -> new int[1],           // container
        (a, b) -> a[0] += b,        // accumulator
        (a, b) -> a[0] += b[0]      // combiner
    )[0];
```

👉 Yaha tumne reduce/Collectors.summingInt ka alternative banaya.

---

### ✅ Scenario 2 (Grouped aggregation)

Normally tum likhte:

```java
Map<String, List<String>> result = employees.stream()
    .collect(Collectors.groupingBy(
        Employee::getDept,
        Collectors.mapping(Employee::getName, Collectors.toList())
    ));
```

Custom collector alternative (rough sketch):

```java
Map<String, List<String>> result = employees.stream()
    .collect(
        HashMap::new,
        (map, e) -> map.computeIfAbsent(e.getDept(), k -> new ArrayList<>()).add(e.getName()),
        (m1, m2) -> {
            m2.forEach((k,v) -> m1.merge(k, v, (l1, l2) -> { l1.addAll(l2); return l1; }));
        }
    );
```

👉 Yaha tumne groupingBy+mapping ke built-in collector ko manually implement kar diya.

## ✅ Using Built-in Collectors

```java
Map<String, Summary> summary = invoices.stream()
    .collect(Collectors.groupingBy(
        Invoice::getCustomer,
        Collectors.collectingAndThen(
            Collectors.toList(),
            list -> {
                int total = list.stream().mapToInt(Invoice::getAmount).sum();
                Set<String> ids = list.stream().map(Invoice::getInvoiceId).collect(Collectors.toSet());
                return new Summary(total, ids);
            }
        )
    ));
```

* Yaha humne `groupingBy + collectingAndThen` + chhoti si logic lagayi.
* Kaam ho gaya, lekin thoda verbose lag raha hai.

## ✅ Custom Collector Alternative

```java
Collector<Invoice, Map<String, Summary>, Map<String, Summary>> customCollector =
    Collector.of(
        HashMap::new, // supplier
        (map, inv) -> {
            map.compute(inv.getCustomer(), (cust, summary) -> {
                if (summary == null) {
                    summary = new Summary(0, new HashSet<>());
                }
                summary.total += inv.getAmount();
                summary.invoiceIds.add(inv.getInvoiceId());
                return summary;
            });
        },
        (map1, map2) -> {
            map2.forEach((cust, summ) -> map1.merge(cust, summ, (s1, s2) -> {
                s1.total += s2.total;
                s1.invoiceIds.addAll(s2.invoiceIds);
                return s1;
            }));
            return map1;
        }
    );

// Usage
Map<String, Summary> result = invoices.stream().collect(customCollector);
```

```java
class Summary {
    int total;
    Set<String> invoiceIds;

    public Summary(int total, Set<String> invoiceIds) {
        this.total = total;
        this.invoiceIds = invoiceIds;
    }
}
```

```java
Collector<Integer, int[], String> statsCollector =
    Collector.of(
        () -> new int[3],                     // [sum, count, avg placeholder]
        (acc, num) -> { acc[0] += num; acc[1]++; },   // accumulate sum & count
        (a1, a2) -> { a1[0]+=a2[0]; a1[1]+=a2[1]; return a1; }, // combine

String result = List.of(10, 20, 30, 40).stream().collect(statsCollector);
```

```java
    static void scenario1GlobalBuiltIn(List<Order> orders) {
        System.out.println("=== Scenario 1 — Global (Built-in) ===");

        double totalRevenue = orders.stream()

    static class Stats {
        long count;
        double sum;
        double min = Double.POSITIVE_INFINITY;
        double max = Double.NEGATIVE_INFINITY;

        void add(double v) {
            count++;
            sum += v;
            min = Math.min(min, v);
            max = Math.max(max, v);
        }
        Stats combine(Stats other) {
            this.count += other.count;
            this.sum += other.sum;
            this.min = Math.min(this.min, other.min);
            this.max = Math.max(this.max, other.max);
            return this;
        }
        double avg() { return count == 0 ? 0.0 : sum / count; }

        @Override public String toString() {
            return String.format("Stats{count=%d, sum=%.2f, min=%.2f, max=%.2f, avg=%.2f}",
                    count, sum, min, max, avg());
        }

        Collector<Order, Map<String, PriorityQueue<Order>>, Map<String, List<String>>> top3PerCustomer =
            Collector.of(
                HashMap::new,

                    pq.offer(o);
                    if (pq.size() > 3) pq.poll();

                (m1, m2) -> {
                    m2.forEach((cust, pq2) -> {
                        PriorityQueue<Order> pq1 = m1.computeIfAbsent(cust, k -> new PriorityQueue<>(byAmountAsc));
                        pq2.forEach(o -> {
                            pq1.offer(o);
                            if (pq1.size() > 3) pq1.poll();
                        });
                    });
                    return m1;

                (map) -> {
                    Map<String, List<String>> out = new HashMap<>();

                        ordersDesc.sort(Comparator.comparingDouble(Order::getAmount).reversed());

                        out.put(cust, ids);
                    });
                    return out;

            );
```

# Collector vs Collectors
Great question 👍 — **`Collector` vs `Collectors`** in Java 8 often confuses people because the names are so similar, but they serve different purposes. Let’s break it down:

---

## 1. **Collector (Interface)**

* **Definition**: `Collector<T, A, R>` is a **functional interface** in `java.util.stream`.
* **Purpose**: It defines a *reduction operation* on elements of a stream (how input elements are accumulated into a result).
* **Generic Types**:

  * `T` → Type of input elements.
  * `A` → Mutable accumulator type.
  * `R` → Result type.
* **Methods**: (important ones)

  * `supplier()`
  * `accumulator()`
  * `combiner()`
  * `finisher()`
  * `characteristics()`
* **Use Case**: You rarely implement `Collector` directly unless you’re creating a **custom collector** (for special reduction logic).

**Example**: Custom collector to concatenate strings with `|`:

```java
Collector<String, StringBuilder, String> customCollector =
    Collector.of(
        StringBuilder::new,
        (sb, str) -> sb.append(str).append("|"),
        (sb1, sb2) -> sb1.append(sb2),
        StringBuilder::toString
    );

String result = Stream.of("A", "B", "C").collect(customCollector);
// Output: A|B|C|
```

---

## 2. **Collectors (Utility Class)**

* **Definition**: `Collectors` is a **final utility class** in `java.util.stream`.
* **Purpose**: Provides a **set of ready-made Collector implementations** so you don’t need to build your own every time.
* **Static Methods**:

  * `toList()`, `toSet()`, `toMap()`
  * `joining()`
  * `groupingBy()`
  * `partitioningBy()`
  * `counting()`, `summarizingInt()`, etc.
* **Use Case**: Whenever you do `collect(Collectors.xyz())`, you’re using one of these predefined collectors.

**Example**:

```java
List<String> names = Stream.of("A", "B", "C")
                           .collect(Collectors.toList());
// Collects into a List
```

---

## 3. **Key Difference Table**

| Aspect       | `Collector` (Interface)           | `Collectors` (Class)                             |
| ------------ | --------------------------------- | ------------------------------------------------ |
| Type         | Functional Interface              | Utility Class                                    |
| Purpose      | Blueprint to define reduction     | Provides ready-made implementations              |
| Who uses it? | Advanced devs (custom collectors) | Almost everyone (everyday use in Streams)        |
| Examples     | `Collector.of(...)`               | `Collectors.toList()`, `Collectors.groupingBy()` |

---

✅ **In short**:

* `Collector` = the contract (interface) for reduction.
* `Collectors` = the toolbox (utility class) full of ready-made collectors.

# So Collector has only 1 method that is of()?

## 🔹 `Collector` (Interface)

`Collector<T, A, R>` is an **interface**.
It does **NOT** have only `of()` method.
It actually defines **5 abstract methods** that every implementation must provide:

1. **`Supplier<A> supplier()`** → creates a new result container (like `new ArrayList<>()`).
2. **`BiConsumer<A, T> accumulator()`** → how to add an element into the container.
3. **`BinaryOperator<A> combiner()`** → how to merge two containers (important in parallel streams).
4. **`Function<A, R> finisher()`** → how to transform the container into the final result (e.g., `StringBuilder → String`).
5. **`Set<Characteristics> characteristics()`** → metadata (like `CONCURRENT`, `UNORDERED`, `IDENTITY_FINISH`).

---

## 🔹 Then where does `of()` come from?

* The `of()` method is **not in the interface**.
* It is a **static factory method in the `Collector` interface** (yes, since Java 8 interfaces can have static methods).
* Purpose: **to easily create custom collectors** without writing a full class.

**Example:**

```java
Collector<String, StringBuilder, String> customCollector =
    Collector.of(
        StringBuilder::new,                // supplier
        StringBuilder::append,             // accumulator
        StringBuilder::append,             // combiner
        StringBuilder::toString            // finisher
    );
```

---

## 🔹 In short

* `Collector` interface = defines the 5 abstract methods.
* `Collector.of(...)` = a **helper static method** to create a `Collector` quickly.
* `Collectors` class = utility class with **lots of prebuilt collectors**.

---

👉 Think like this:

* **`Collector`** = contract.
* **`Collector.of()`** = DIY kit to make your own collector.
* **`Collectors`** = ready-made tools from the factory.

# Spliterator
Good question bhai 👌 — **Spliterator** is one of the hidden gems of Java 8. Let me break it down clearly for you:

---

## 🔹 What is a Spliterator?

* **Spliterator** = *Split + Iterator*
* It’s like an advanced **Iterator**, introduced in **Java 8**, to **traverse and partition** elements of a source (like Collection, Stream, Array, IO channel).
* Unlike a normal `Iterator`, which only moves forward one-by-one, a **Spliterator** can **split** the elements into parts so they can be **processed in parallel**.

---

## 🔹 Why Spliterator?

* Iterator was good for sequential traversal.
* But Java 8 Streams introduced **parallel processing**, so we needed a **smarter iterator**.
* Spliterator helps in **efficient parallelism** by dividing data chunks and distributing them across multiple threads.

---

## 🔹 Key Methods of Spliterator

```java
boolean tryAdvance(Consumer<? super T> action);
Spliterator<T> trySplit();
long estimateSize();
int characteristics();
```

1. **tryAdvance(Consumer action)** → like `Iterator.next()`, it processes one element at a time.
2. **trySplit()** → tries to split the Spliterator into two parts (for parallel work).
3. **estimateSize()** → gives estimated number of elements remaining.
4. **characteristics()** → returns characteristics (ORDERED, SORTED, SIZED, DISTINCT, IMMUTABLE, etc).

---

## 🔹 Example Usage

```java
import java.util.*;
import java.util.stream.*;

public class SpliteratorDemo {
    public static void main(String[] args) {
        List<String> names = Arrays.asList("Danish", "Aman", "Ravi", "Pooja", "Neha");

        Spliterator<String> spliterator = names.spliterator();

        // Sequential traversal
        System.out.println("Sequential Traversal:");
        spliterator.forEachRemaining(System.out::println);

        // Splitting for parallel
        Spliterator<String> s1 = names.spliterator();
        Spliterator<String> s2 = s1.trySplit();

        System.out.println("\nPart 1:");
        s1.forEachRemaining(System.out::println);

        System.out.println("\nPart 2:");
        if (s2 != null) {
            s2.forEachRemaining(System.out::println);
        }
    }
}
```


## 🔹 When to Use Spliterator?

* When you need **parallel processing** with Streams.
* When writing your own **custom data source** for streams.
* When you need **fine control** over iteration + splitting behavior.

---

✅ **In short**:
Iterator → sequential, simple.
Spliterator → smart, can split + iterate, supports parallel streams.

# 2) Real insight: Custom **LoggingSpliterator**

Source ke andar kya ho raha hai (split vs advance) ye dekhne ka best dev-way: apna wrapper Spliterator banao jo **`trySplit()`**, **`tryAdvance()`**, **`estimateSize()`** ko log kare.

```java
import java.util.Spliterator;
import java.util.function.Consumer;

class LoggingSpliterator<T> implements Spliterator<T> {
    private final Spliterator<T> delegate;
    private final String name;

    LoggingSpliterator(Spliterator<T> delegate, String name) {
        this.delegate = delegate;
        this.name = name;
    }

    @Override
    public boolean tryAdvance(Consumer<? super T> action) {
        return delegate.tryAdvance(e -> {
            System.out.println("[" + name + "] tryAdvance: " + e);
            action.accept(e);
        });
    }

    @Override
    public Spliterator<T> trySplit() {
        Spliterator<T> split = delegate.trySplit();
        if (split != null) {
            System.out.println("[" + name + "] trySplit → new part created");
            return new LoggingSpliterator<>(split, name + ".split");
        } else {
            System.out.println("[" + name + "] trySplit → null (no more split)");
            return null;
        }
    }

    @Override public long estimateSize() {
        long s = delegate.estimateSize();
        System.out.println("[" + name + "] estimateSize: " + s);
        return s;
    }

    @Override public int characteristics() { return delegate.characteristics(); }
}
```

# Parallel Stream
Chintu mode ON 😎
Bhai, **Parallel Stream ka full post-mortem** yahi pe. Clear, structured, and with gotchas + best practices. Ready?

# 1) Parallel Stream hai kya?

* **Goal:** Ek hi stream ka kaam multiple threads me tod-fod karke jaldi finish karna.
* **Engine:** `ForkJoinPool.commonPool()` (by default ~CPU cores ke barabar threads).
* **Kaise kaam karta hai (high level):**

  1. **Splitting:** Source ka `Spliterator` data ko chhote chunks me `trySplit()` karke todta hai.
  2. **Processing:** Har chunk alag worker thread me pipeline (map/filter/reduce…) chalaata hai.
  3. **Combining:** Partial results ko **combiner** se merge karta hai.

# 2) Kab use kare? (Rule of Thumb)

Use **tab** karo jab:

* **CPU-bound** heavy computation ho (hashing, parsing, math, crypto, JSON decode, etc.).
* **Large data (10^5+ items)** aur **parallel-friendly source** (array, `ArrayList`, `IntStream.range`, `Files.lines` with care).
* **Stateless operations** (map/filter) + **associative reduction** (sum/min/max, grouping by).

Avoid **tab** karo jab:

* **I/O-bound** (DB calls, HTTP calls) → threads idle rahenge, context switching badhega.
* **Small data** / light ops → overhead > benefit.
* **Order-sensitive** pipelines (e.g., `sorted().limit(…)`, `findFirst`) → speedup kill ho jata.
* **Shared mutable state** touched in `forEach` → race conditions.

# 3) Source/Spliterator ki “parallel-friendliness”

Best sources (fast split + balanced chunks):

* **Arrays**, `ArrayList`, `IntStream.range(…)`, `LongStream.range(…)`
* Inke Spliterator me **SIZED**, **SUBSIZED**, **ORDERED** (ArrayList) hote hain → perfect splitting.

Poor sources:

* **LinkedList**, **synchronized collections**, **Iterator-based** sources, **Stream.generate/iterate** (unless `iterate` ke naye overload ke saath `takeWhile` etc.).
* Heavy **boxing/unboxing** (use primitive streams).

Tip: Agar order ki zarurat **nahi**, to `.unordered()` lagao → merge cheap hota hai.

# 4) Semantics: map/filter/reduce parallel me kaise behave karte?

* **map/filter**: har chunk thread-local apply hota, koi issue nahi (agar side-effects nahi).
* **reduce / collect**:

  * Associativity **must**: `(a ⊕ b) ⊕ c == a ⊕ (b ⊕ c)` (e.g., sum, min, set union).
  * **Combiner** sahi likhna zaroori: parallel me bahut dafa call hota.
* **Short-circuit** ops (`anyMatch`, `findAny`) parallel me chamakte hain. `findFirst` mehnga (order preserve).

# 5) ForkJoinPool control & thread count

* Default: `ForkJoinPool.commonPool()` use hota.
* **Custom size per pipeline** chahiye? Pattern:

  ```java
  ForkJoinPool pool = new ForkJoinPool(8);
  List<Integer> data = IntStream.range(0, 1_000_000).boxed().toList();
  int sum = pool.submit(() -> data.parallelStream().mapToInt(Integer::intValue).sum()).join();
  ```
* Global tweak (early in app):
  `System.setProperty("java.util.concurrent.ForkJoinPool.common.parallelism", "8");`
  (App start par set karo, varna late hone par effect nahi.)

# 6) Thread-safety & Side-effects (ye sabse bada pitfall)

**DON’Ts**

```java
List<Integer> out = new ArrayList<>();
list.parallelStream().forEach(out::add); // RACE + undefined order
```

**DOs**

* Collectors use karo:

  ```java
  List<Integer> out = list.parallelStream().collect(Collectors.toList()); // thread-safe via internal merging
  ```
* Aggregation ke liye **associative** reducers:

  ```java
  int sum = list.parallelStream().mapToInt(i -> i).sum();
  ```

# 7) Ordering ke costs

* **ORDERED** stream me merge/limit/sorted mehnga.
* Agar order irrelevant ho:

  ```java
  stream.unordered().parallel().forEach(...);
  ```
* `findAny()` > `findFirst()` parallel me.
* `limit(n)` parallel+ordered me slow ho sakta; unordered + `limit` better (agar logically allowed).

# 8) Collectors: concurrent vs normal

* **Normal:** `groupingBy`, `toList`, `toSet` → parallel me internally buckets merge honge (OK but not fully concurrent).
* **Concurrent:** `groupingByConcurrent`, `toConcurrentMap` → **CONCURRENT** collector; unordered me extra fast.

  ```java
  Map<String, Long> counts =
      words.parallelStream()
           .collect(Collectors.groupingByConcurrent(w -> w, Collectors.counting()));
  ```

# 9) Code Patterns (good vs bad)

### ✅ Good: CPU-heavy compute

```java
double result =
    IntStream.range(0, 10_000_000)
             .parallel()
             .mapToDouble(x -> Math.sin(x) * Math.cos(x))
             .sum();
```

### ✅ Good: Parallel grouping (unordered OK)

```java
Map<Character, Long> freq =
    text.chars().parallel()
        .mapToObj(c -> (char) c)
        .unordered()
        .collect(Collectors.groupingByConcurrent(c -> c, Collectors.counting()));
```

### ❌ Bad: Shared mutable state

```java
AtomicInteger ai = new AtomicInteger();
list.parallelStream().forEach(x -> ai.addAndGet(expensive(x))); // works but often slower than reduce
```

Prefer:

```java
int total = list.parallelStream().mapToInt(x -> expensive(x)).sum();
```

### ❌ Bad: IO in parallel stream

```java
urls.parallelStream().map(this::httpGet).toList(); // Thread starvation / timeouts risk
```

Better: **Bounded** thread pool + async HTTP client.

# 10) Performance checklist (quick mind-map)

1. **Data size:** <50k? Try sequential first; measure.
2. **CPU-bound?** Yes → parallel try. No → avoid.
3. **Source splittable?** Array/ArrayList/IntStream.range = 👍
4. **Order needed?** If no → `.unordered()`
5. **Ops stateless?** Yes → 👍
6. **Reduction associative?** Yes → 👍
7. **Boxing avoid?** Use primitive streams (`IntStream`, `LongStream`).
8. **Measure!** Warm-up (JIT), multiple runs, average.

# 11) Micro-bench (drop-in snippet)

> Isse tum apne machine par quick feel le sakte ho.

```java
import java.util.*;
import java.util.concurrent.*;
import java.util.stream.*;

public class ParBench {
  public static void main(String[] args) {
    int N = 20_000_000;
    int reps = 3;

    long bestSeq = Long.MAX_VALUE, bestPar = Long.MAX_VALUE;

    for (int r = 0; r < reps; r++) {
      long t1 = System.nanoTime();
      long s1 = IntStream.range(0, N)
                         .map(i -> i * 31 ^ 0x9e3779b9)
                         .filter(x -> (x & 1) == 0)
                         .asLongStream().sum();
      long t2 = System.nanoTime();

      long t3 = System.nanoTime();
      long s2 = IntStream.range(0, N).parallel()
                         .map(i -> i * 31 ^ 0x9e3779b9)
                         .filter(x -> (x & 1) == 0)
                         .asLongStream().sum();
      long t4 = System.nanoTime();

      bestSeq = Math.min(bestSeq, t2 - t1);
      bestPar = Math.min(bestPar, t4 - t3);

      if (s1 != s2) throw new AssertionError("Results differ!");
    }

    System.out.printf("Best Seq: %.2f ms%n", bestSeq / 1e6);
    System.out.printf("Best Par: %.2f ms%n", bestPar / 1e6);
    System.out.printf("Speedup:  %.2fx%n", (double)bestSeq / bestPar);
  }

```

# 12) Spliterator angle (kyunki tum pooch chuke ho)

* **Role:** Parallelization ka foundation. `trySplit()` se roughly half-half chunks banata.
* **Characteristics (important):**

  * **SIZED/SUBSIZED** → perfect splitting (arrays/ArrayList).
  * **ORDERED** → order preserve karna mehnga.
  * **CONCURRENT** → source me concurrent modification allowed.
  * **IMMUTABLE/NONNULL** hints → optimizations.
* **Custom Spliterator** likh ke tum apni domain sequence ko parallel-friendly bana sakte ho.

# 13) Common pitfalls (interview + real life)

* **Non-associative reduction:** floating-point subtraction/division, string concat in wrong place.
* **`sorted()` + `parallel()`**: heavy; kabhi kabhi `Arrays.parallelSort` ya map-reduce pattern better.
* **`limit()` with ORDERED:** performance nalla; unordered ho to theek.
* **Exceptions:** multi-thread stack traces messy; wrap carefully.
* **Parallel stream everywhere:** anti-pattern; pehle **measure**, phir enable.

# 14) Concurrent collectors quick reference

* **Faster (unordered):** `groupingByConcurrent`, `toConcurrentMap`, `mapping` + concurrent.
* **Normal but OK:** `groupingBy` (merge phase hota).
* **Lists/Sets:** `toList()` Java 16+ returns unmodifiable list; parallel me internally safe.
  If truly concurrent mutate chahiye: `toCollection(CopyOnWriteArrayList::new)` (but heavy), or collect then publish.

# ForkJoinPool
Bahut sahi catch kiya bhai 👌
Direct answer:

* **Sequential stream** → **ForkJoinPool use nahi hota**. Ye normal ek hi thread (tumhara main thread) me chalta hai.
* **Parallel stream** → **ForkJoinPool (commonPool)** use hota hai, jisme multiple worker threads parallel me kaam karte hain.

---

### Thoda deep me:

1. **Sequential Stream:**

   ```java
   list.stream().map(x -> x * 2).toList();
   ```

   * Ye pipeline **tumhare current thread** me sequentially execute hoti hai.
   * Koi extra thread pool, koi ForkJoinPool involve nahi.

2. **Parallel Stream:**

   ```java
   list.parallelStream().map(x -> x * 2).toList();
   ```

   * Ye source ka data todta hai (`Spliterator.trySplit()`).
   * Subtasks **ForkJoinPool.commonPool()** ke threads me bheje jaate hain.
   * Har worker thread apna chunk process karta hai.
   * Last me results combine (reduce/collect).

### Extra point (interview material 😉):

* **ForkJoinPool ka asli purpose** parallel stream ke alawa bhi hai:

  * Recursive algorithms (merge sort, quick sort, Fibonacci etc.) me tod–fod kar ke parallel run karna.
  * Example: `RecursiveTask` / `RecursiveAction`.
* Lekin **stream API ke context me** → sequential stream = no pool, parallel stream = ForkJoinPool.

# 1) Parallel Sum with `RecursiveTask<Long>`

```java
import java.util.concurrent.*;

public class ParallelSum extends RecursiveTask<Long> {
  private static final int THRESHOLD = 50_000; // is se chhote chunks sequential kar do
  private final long start, end;

  public ParallelSum(long start, long end) {
    this.start = start;
    this.end = end;
  }

  @Override
  protected Long compute() {
    long length = end - start + 1;
    if (length <= THRESHOLD) {
      long sum = 0;
      for (long i = start; i <= end; i++) sum += i;
      return sum;
    }

    long mid = (start + end) >>> 1;
    ParallelSum left = new ParallelSum(start, mid);
    ParallelSum right = new ParallelSum(mid + 1, end);

    // fork left, compute right (work-stealing friendly)
    left.fork();
    long rightAns = right.compute();
    long leftAns = left.join();

    return leftAns + rightAns;
  }

  public static void main(String[] args) {
    long N = 1_000_000_000L; // 1e9
    ForkJoinPool pool = new ForkJoinPool(); // default = available cores

    long t1 = System.nanoTime();
    long ans = pool.invoke(new ParallelSum(1, N));
    long t2 = System.nanoTime();

    // sequential for comparison
    long t3 = System.nanoTime();
    long seq = 0;
    for (long i = 1; i <= N; i++) seq += i;
    long t4 = System.nanoTime();

    System.out.println("Parallel Sum = " + ans + " in " + (t2 - t1)/1e6 + " ms");
    System.out.println("Sequential   = " + seq + " in " + (t4 - t3)/1e6 + " ms");
  }
}
```

**Samjho:**

* Bada range → half-half tod rahe (divide & conquer).
* `left.fork()` task queue me chala gaya; current thread `right.compute()` karta.
* Baad me `left.join()` se result le liya (work-stealing model help karta).

# 2) Parallel Merge Sort with `RecursiveAction`

```java
import java.util.*;
import java.util.concurrent.*;

public class ParallelMergeSort extends RecursiveAction {
  private static final int THRESHOLD = 10_000; // chhote arrays ko sequential sort
  private final int[] arr;
  private final int left, right;

  public ParallelMergeSort(int[] arr, int left, int right) {
    this.arr = arr; this.left = left; this.right = right;
  }

  @Override
  protected void compute() {
    int len = right - left + 1;
    if (len <= THRESHOLD) {
      Arrays.sort(arr, left, right + 1);
      return;
    }

    int mid = (left + right) >>> 1;
    ParallelMergeSort leftTask = new ParallelMergeSort(arr, left, mid);
    ParallelMergeSort rightTask = new ParallelMergeSort(arr, mid + 1, right);

    invokeAll(leftTask, rightTask); // fork both & join automatically
    merge(arr, left, mid, right);
  }

  private static void merge(int[] a, int l, int m, int r) {
    int[] tmp = new int[r - l + 1];
    int i = l, j = m + 1, k = 0;
    while (i <= m && j <= r) tmp[k++] = (a[i] <= a[j]) ? a[i++] : a[j++];
    while (i <= m) tmp[k++] = a[i++];
    while (j <= r) tmp[k++] = a[j++];
    System.arraycopy(tmp, 0, a, l, tmp.length);
  }

  public static void main(String[] args) {
    int n = 5_000_000;
    int[] a1 = new Random(42).ints(n).toArray();
    int[] a2 = Arrays.copyOf(a1, a1.length);

    ForkJoinPool pool = new ForkJoinPool();

    long t1 = System.nanoTime();
    pool.invoke(new ParallelMergeSort(a1, 0, a1.length - 1));
    long t2 = System.nanoTime();

    long t3 = System.nanoTime();
    Arrays.sort(a2);
    long t4 = System.nanoTime();

    System.out.printf("Parallel MergeSort: %.2f ms%n", (t2 - t1)/1e6);
    System.out.printf("Sequential sort:    %.2f ms%n", (t4 - t3)/1e6);

    // sanity check
    System.out.println("Equal? " + Arrays.equals(a1, a2));
  }
}
```

**Key points:**

* `RecursiveAction` return value nahi deta (sirf action).
* `invokeAll` = do tasks **fork** + **join** efficiently.
* Threshold set karke oversplitting se bacho (warna overhead > benefit).

---

## Quick Tips (yaad rakhna):

* **Threshold** ko workload & machine ke hisaab se tune karo.
* CPU-bound heavy kaam me ForkJoinPool chamakta; I/O-bound me nahi.
* Parallel Stream bhi **andar hi andar isi pattern** ko follow karta (split → fork → compute → join → combine).

# Java 8 `Optional` – Complete Interview Preparation Guide

`Optional` was introduced in **Java 8** to solve one of the biggest problems in Java:

# 👉 `NullPointerException (NPE)`

Before Java 8:

```java
String name = null;
System.out.println(name.length()); // NullPointerException
```

Developers had to write too many null checks:

```java
if(name != null){
    System.out.println(name.length());
}
```

To reduce this problem and write cleaner code, Java introduced:

# ✅ `Optional<T>`

---

# 1. What is Optional?

`Optional` is a **container object** that may or may not contain a value.

Think like:

| Situation     | Optional State     |
| ------------- | ------------------ |
| Value present | Non-empty Optional |
| Value absent  | Empty Optional     |

---

# 2. Real World Analogy

Imagine Swiggy order tracking.

```text

- Delivery Boy
OR
- No Delivery Boy assigned yet
```

Instead of returning `null`:

```java
DeliveryBoy boy = null;
```

Return:

```java
Optional<DeliveryBoy>
```

This clearly tells developers:

# “Value may be absent. Handle carefully.”

```java
Optional<String> name = Optional.of("Danish");
```
